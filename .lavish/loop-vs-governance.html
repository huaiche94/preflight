定位圖解 Loop engineering ／ harness　vs　Auspex

誰「跑」loop，誰「治理」loop

大家嘴上講的 loop engineering 指的是「跑那條 agentic loop」的程式； Auspex 是另一層——它不跑 loop，而是站在 loop 之外治理一條它並不擁有的 loop。 詞彙重疊，軸線不同。下面用一張圖 + 一個例子把差別攤開。

A · Harness　跑 loop　·　擁有 loop
B · Auspex　治理 loop　·　不擁有 loop

① 一張圖：B 環繞 A

內圈（藍）是 harness 自己在跑的 model → tool → 讀結果 → 再決定 迴圈； 外圈（金）是 Auspex 的三個治理接觸點——翻頁前、翻頁後、配額撞牆。 Auspex 從不進到迴圈裡面，只在邊界上把關。

藍＝A 擁有的內層迴圈（就是你之前推導的「切片內 red→green 自我修正」那條）。金＝B 在邊界上的閘門。

② 一個例子走一遍：「為 /login 加上 rate limiter」

同一個任務，時間由上往下。左欄看 A 在迴圈裡做什麼，右欄看 B 在迴圈外做什麼。 注意 B 一次都沒踏進迴圈——它只在「翻頁前 / 翻頁後 / 配額見底」三個時刻出手。

時刻 A · Harness 跑 loop（擁有 loop） B · Auspex（loop 之外治理）
翻頁前 —（迴圈還沒開始） forecast：這輪約 4.8k tokens、配額還夠 → policy 判 RUN；
CHECKPOINT_AND_RUN 先把 repo 狀態固化，翻壞了能退。
迴圈第 1–4 圈 讀 login handler → 插入 rate-limit middleware → 跑測試 🔴 紅 → 修 → 再跑 🟢 綠。
這圈就是內層 micro-loop，A 自己收斂。		 不介入迴圈內部；只在旁監看 token / 配額的燃燒速率。
翻頁後 宣稱「rate limiter 做好了」。 evidence gate：要求測試產物 + git 快照才算數；
拿不出證據 → 擋住，不標記完成（防壓縮後偷偷送 regression）。
凌晨 2:00 配額見底 若硬跑，迴圈會死在半途，白燒一整夜到你早上發現。 Graceful Pause：找安全點 → checkpoint → 寫 wake task 進 SQLite；
天亮配額回來，daemon 重驗狀態後自動恢復，不是從頭來。

重點：紅→綠那條自我修正迴圈，A 全程自理；B 只在邊界（翻頁前後、配額）出手，且從不決定 A 的下一步。

③ A 與 B，並排看

A · Harness ／ loop engineering

做什麼
跑 agentic loop：派發工具 → 讀結果 → 決定下一步 → 重複到完成；＋維持 loop 的 context 工作（壓縮、子 agent、停止條件）。
代表
Claude Code、Claude Agent SDK、LangGraph。
跟 loop 的關係
擁有那條 loop —— 它自己就是 loop。
一句話：讓一條 loop 跑起來。

B · Auspex

做什麼
跑在 loop 之外治理它：forecast → policy gate → checkpoint → evidence gate → 配額撞牆轉暫停。
代表
Auspex —— Go 單一 binary、本機、SQLite，無雲端。
跟 loop 的關係
不擁有、環繞它；provider-agnostic（Claude Code / Codex CLI 都能罩）。
一句話：讓一條它不擁有的 loop 可問責。
🎯
同一組詞，不同軸線。harness、loop、checkpoint、verify —— A 和 B 都在用，但 A 讓 loop 跑起來、B 讓 loop 可問責。 別把 B 講成 A：「跑 loop」那條賽道已被 Claude Code / SDK / Cursor 塞滿； Auspex 要守的護城河是治理 ／ 可觀測這一條——尤其是沒人在 loop engineering 裡談的 forecast ／ 配額治理。
