## 整體回顧：checkpoint 角色的完整歷程（Wave 1 - Wave 9，checkpoint-a01 至 checkpoint-b09）

這是此角色最後被指派的 DAG 節點（`checkpoint-b09`）；`EXECUTION_DAG.md`
中不再有任何 `checkpoint-*` 節點。九個波次（a01-a09、b01-b09）共出貨了
十九個節點，再加上一項修正性修補（`corrective-qa05-p1`）。有幾個模式在
整個歷程中一致到值得在這個高度上完整記錄一次，而不只是逐節點記錄：

1. **兩個可獨立測試的子元件，從頭到尾都維持可獨立測試。** Part A
   （`internal/progress`、`internal/statecheckpoint`、
   `internal/artifacts`）與 Part B（`internal/gitx`、
   `internal/repocheckpoint`、`internal/redact`）從未需要互相匯入，完全
   符合 `agents/checkpoint.md`「Cross-part boundary」一節從第一天就規定
   的要求。這並非沒有代價 —— 它需要刻意設計的注入式介面接縫
   （`statecheckpoint` 中的 `TreeReader`、`progress` 中的
   `EventPublisher` 接縫、`repocheckpoint` 中的
   `resolveWorktree`/`WorktreeLocation` callback），而不是更方便的直接
   匯入 —— 但它在 `checkpoint-a09` 中直接回本：要證明兩個 Reconciler 對
   同一個當機視窗達成一致，正是仰賴它們是兩個真正分開、獨立建置的邏輯，
   而不是一個可能靜默地自我一致的共用抽象。

2. **前一個節點自身的範圍邊界，可靠地預測了下一個節點真正、窄小的
   工作 —— 共出現五次**（`b04`->`b05`、`b06`->`b07`、`a04`->`a05`、
   `a05`->`a08`，以及 `a04`/`a06`->`a09` 自身「證明各部件能組合」的
   框架）。每一個這類「驗證還剩下什麼」的節點，一旦從前一個節點自身的
   筆記正確辨識出真正範圍，就會落在其名目規模以內或以下，而不是從一張
   白紙獨立重新推導。此教訓可一般化為：在假設需要一個全新設計階段之
   前，先把緊接在前的節點自身的 lessons-learned／假設一節，當成一個字
   面的範圍預測來讀。

3. **同一種 fixture 品質的失敗模式，在整個歷程中重複出現三次**：
   `checkpoint-a04` 測試替身 `seqIDGenerator` 的競態（Wave 6）、
   `checkpoint-b06` 過短的合成 Azure-key fixture（Wave 6），以及
   `checkpoint-a09` 跨節點內容相同的 bug（Wave 9，即本文件自身上方的那
   一列）。這三者本質上是同一個錯誤 —— 一個太一致、太容易、或不具併發
   安全性的 fixture，會靜默地對測試自稱要證明的那個確切不變量做不足的
   測試，而測試仍然顯示綠燈（或者，如 a09 的情況，因為錯誤的理由失
   敗，等於自己抓到了自己）。建議把這一點從「學過三次的教訓」提升為一
   個具名、明確的審查問題，供此程式庫中任何未來角色設計 fixture 時使
   用：「一個多實體測試中的每一個實體，是否真的在被測試的維度上，與它
   的手足有所不同；每一個測試替身，在真正正式程式碼所支援的併發模式
   下，是否真的安全？」

4. **在信任一項設計之前先做實驗驗證，而不只是程式碼審查，找出了此角
   色每一個不明顯的 bug** —— `seqIDGenerator` 的競態（由 `-race` 找
   到，不是靠閱讀）、`git apply --check` 的 base-state 語意（由暫存
   repo 重現找到，不是靠閱讀 ADD）、Azure-key fixture（由一個可拋棄的
   檢查腳本找到），以及最舉足輕重的、`checkpoint-b09` 在 `Verify` 中的
   manifest 路徑穿越弱點（由撰寫並執行一個真的讀到 `ArtifactRoot` 之外
   某檔案的獨立概念驗證找到，不是靠推理「這段程式碼看起來有風
   險」）。這是此角色最強而有力、反覆出現的做法，值得為接下來無論哪個
   角色或波次明白地寫下來：當一項設計或一項安全性質可以透過實際執行某
   個東西來驗證時，就去執行它 —— 在斷定「這沒問題」或「這壞了」之前。

5. **Part A／Part B 切分下的兩個「最終關卡」節點，最後被證明有真正不
   同的重心**，與各自 DAG 的風險註記完全吻合：`checkpoint-a09`（整合
   風險）找到零個正式程式碼 bug，產出的是既有正確組合的純粹證明；
   `checkpoint-b09`（安全風險）找到一個真實、先前未知、可被利用的弱
   點，並在證明能夠誠實成立之前，需要一次真正的正式修正。兩種結果，相
   對於各自的風險框架而言都是正當的 —— 整合關卡的工作是證明組合成
   立，它可以誠實地做出「一切早已正確」的結論；安全關卡的工作是找出哪
   裡有問題，一個什麼都沒找到的安全關卡節點，應該把這件事當成要更用力
   稽核的理由，而不是安全無虞的確認。`checkpoint-b09` 一旦開始找到真正
   的缺口，就沒有停在 DAG 自身指名的必要測試類別上 —— 它持續稽核其擁
   有 package 中的每一個路徑拼接，直到用盡合理的不受信任輸入表面為
   止，這正是此角色會建議未來任何具有同等風險註記的節點所應採取的安全
   關卡姿態。

除了上方每一節點已經記載的內容之外，沒有進一步的建議；本節的存在，只
是為了指名整個歷程中反覆出現的模式，而不是任何單一節點內部的模式。

6. **「每一個個別部件都經過測試」與「凍結的介面有一個真正的實作」，
   並非同一項主張，而此角色自身 a01-a09 的檢查清單中，沒有任何一項會
   抓到這個差異。** 最終整合關卡發現，`app.ProgressTreeService` ——
   `cmd/auspex/main.go` 用來組合應用程式根節點所需的真正 contract ——
   在整個儲存庫中沒有任何具體型別實作它，儘管每一個底層部件
   （`NodeStore`、`EdgeStore`、`ArtifactStore`、狀態機、`CompleteNode`、
   `Reconciler`）都已在九個波次中被詳盡測試過，包括 `checkpoint-a09`
   自身「整個堆疊端到端撐得住」的整合關卡。`a09` 證明了各部件彼此能夠
   組合；它從未問過「是否有任何東西滿足外部呼叫者會使用的、真正凍結的
   Go 介面」。這個缺口之所以能在一個明確設計來抓出組合問題的整合關卡
   中存活下來，是因為該關卡自身的測試鷹架（`newFullStackHarness`、
   `complete_node_integration_test.go`）是在測試內部手動把各部件串接起
   來，這證明了各部件「能夠」組合，但這與一個「確實」在凍結 port 之後
   組合它們的正式型別，並不是同一種產出物。給任何未來在其 DAG 末端有
   一個凍結介面的多部件角色的建議：新增一個明確、具名的檢查清單項目
   —— 與「各部件是否能一起運作」分開 —— 檢查「是否存在一個
   `var _ app.X = (*ConcreteType)(nil)` 斷言，且是針對一個真正的正式型
   別，而不只是一個測試替身」。一個角色可以通過它自己每一個 DAG 節
   點，卻仍然留下這個缺口，因為沒有任何單一節點的簡報恰好直接問到這個
   問題。
7. **修正本身證實了 DAG 自身反覆的框架設定是對的：這是組合，不是新邏
   輯。** `internal/progress.Service`（此次修正性新增）約 250 行，其中
   沒有任何一行重新實作一次狀態轉換、一項原子性保證、或一項
   idempotency 檢查 —— 每一個方法都是 DTO 轉譯，加上一次對已經測試過
   的部件的直接呼叫。唯一實質的判斷，是 `FailNode` 的 `FailureClass`：
   凍結的 `FailNodeRequest` DTO 接受這個欄位，但此角色擁有的
   `0020-0029` migration 範圍中沒有任何持久化的欄位對應它；沒有靜默地
   丟棄它，也沒有在此次修正性新增聲明的範圍之外發明一個欄位，而是驗證
   它非空，並記載為一個明確、範圍狹窄、留給未來節點的缺口 —— 與此角色
   Constitution 義務（§7.3）早已要求、針對 provider 能力缺口的「明確揭
   露缺口，而不是掩蓋它」紀律相同，只是這次套用在一個介面／schema 缺
   口上。
