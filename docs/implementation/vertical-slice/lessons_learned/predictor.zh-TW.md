## 跨節點觀察

- 三個節點的實際檔案數都等於或低於 DAG 的預估（3/3、4/5、3/4）——在這個 phase 中，對於一個自成一體、依賴很輕的套件（沒有 I/O、沒有並行處理、除了 `internal/domain` 之外沒有跨套件連接），「M」複雜度傾向於高估大約一個檔案的數量。這是一個小樣本（n=3），在有更多 phase 的資料之前，不應該過度推廣成一個全域的校準修正。
- 遇到的唯一一個流程層級「阻塞項」（一次 session 中途中斷）並不是 DAG 意義下的任務阻塞項——它是靠協調者用 build/test 指令重新驗證磁碟上的實際狀態來解決的，而不是相信對話記憶，這正是 Progress Tree 的哲學（Constitution §6 第 1 條規則：「對話上下文……絕不是真理來源」）套用在更高一層——協調者如何監督一個角色的 session。值得記下來，作為 Auspex 自身價值主張的一個正面案例研究：一次經過 build/test 驗證的恢復，避免了在無聲無息中遺失或重複工作。
- 沒有任何阻塞項、非預期依賴或範疇意外嚴重到需要提出一份 ADR 或偏離已凍結的合約。`internal/domain` 和 `internal/app/ports.go` 已經具備了所有需要的形狀（Confidence、RunwayForecast、Task、ProgressNode），完全不需要請求新增。

### predictor-08 冷啟動不變量驗證做法

DAG 對這個節點自己標註的風險提示（「高風險——絕不能把未校準的分數標成機率」），加上任務指示明確要求「一套專門、明確的測試套件，證明冷啟動/未校準絕不會變成機率這個不變量，在每一個政策行動上都成立」，促成了一套兩層驗證策略，而不只是多寫幾個測試案例：

1. **結構面**：整個套件中恰好只有兩處呼叫點（`decide.go` 的 `runwayPauseDecision`，兩處都在賦值前緊接著一個明確的 `rf.Calibrated &&` 檢查）會建構出非 nil 的 `Decision.Probability`。套件中其他每一個 Decision 字面值都把它設為字面上的 `nil`。這是可以透過檢視/grep 驗證的，而不只是靠跑測試——未來如果有人想從另一條程式碼路徑偷渡一個機率值進來，只要一行 diff 之差就會打破一個已記錄、具名的不變量（`Decision.Probability` 自己的文件註解就指名了確切的防護），而不會是一次無聲的行為漂移。
2. **行為面**：`coldstart_test.go` 證明的是「範圍較窄但正確」的主張（「未校準絕不會變成機率」），而不是「範圍較廣但錯誤」的主張（「機率永遠是 nil」），做法是刻意加入一個對照組案例（`TestColdStartArmedButNotYetConfirmedRunwayIsCalibratedAndMayReportProbability`），在這個案例中，一個真正經過校準的 Runway 輸入，正當地產生了一個非 nil 的 Probability。建議把這種「證明較窄的主張，並為較廣但錯誤的主張加入一個對照組案例」的模式，用在未來任何守護一個由布林值控制欄位的 Auspex 不變量測試套件上——一套只會在各處斷言「這個欄位是 nil」的套件，即使把控制邏輯整個刪掉也照樣會通過，會在測試依然全綠的情況下，無聲無息地讓這個不變量變得毫無意義。

## Wave 7 跨節點觀察

- 這是第一個在單一 `app.TxRunner.WithTx` transaction 裡，跨整個多階段 pipeline 呼叫持久化真實狀態的 predictor 節點（feature_vectors + predictions + policy_decisions 作為一次原子性寫入）——這呼應了 `checkpoint` 的 `ProgressTreeService.CompleteNode` 紀律（CONTRACT_FREEZE.md 的 transaction-boundary 章節），只是套用在另一個角色套件的更高一層，證實了這項紀律可以乾淨地套用到「大量讀取、最後一次寫入」的 pipeline 形狀，而不只是單一實體的更新。
- `ConsumeAuthorization` 的恰好一次（exactly-once）保證，是靠一個單一、有條件的 `UPDATE ... WHERE consumed_at IS NULL`，在跟驗證綁定/過期的讀取「同一個」transaction 裡強制執行的，而不是靠應用層級的「先檢查再寫入」模式——後者在並行呼叫者之下存在一個 TOCTOU 競態，而前者則從結構上把它封住了。這一點是透過一個專門的並行 goroutine 測試直接驗證的（`TestConsumeAuthorization_ConcurrentReplayOnlyOneWins`，8 個 goroutine 競爭同一個 authorization ID，斷言恰好只有 1 次成功），這是在 `go test -race` 之外「額外」做的，因為單靠 `-race` 只能證明沒有 data race，不能證明沒有 logic race（兩個 goroutine 可以各自乾淨地取得自己的 transaction，如果 UPDATE 的 WHERE 子句漏掉了，兩者仍然可能都「成功」）——建議把這個區別（data race 對比 logic race，以及單靠 `-race` 抓不到後者）明確地點出來，用在這個專案未來任何恰好一次/冪等性節點上。
- 這個節點測試中每一個固定時間點的斷言（過期邊界、確定性輸出）都是靠一個本地的 `fakeClock`（帶有明確的 `Advance` 方法）驅動的，絕不使用 `time.Now()`——延續了 `internal/scheduler/lease_test.go` 早已建立的專案全域慣例，證實了這個模式（而不是一個共享的、跨套件的 fake）即使對一個比之前多數 predictor 節點都更依賴時鐘（過期判斷）的節點來說，仍然是正確的預設做法。

## Wave 8 跨節點觀察

- 一個稽核/強化型節點（重新驗證既有、早已上線的邏輯，而不是建構新邏輯）跟一個建構型節點的成本結構不同：幾乎所有的心力都花在判斷哪些對抗性情境值得測試、以及夠仔細地閱讀既有程式碼/文件/歷史，才能找出那唯一一個真實的不對稱（prompt-hash 綁定是依 request 而定，不是依那一列資料而定），而一旦缺口被找出來，修正本身和多數新測試都是機械性的。建議 DAG 繼續明確區分「稽核」節點和「建構」節點（就像 predictor-10 已經做的那樣，透過它專屬的 `-run Authorization` 關卡，以及 predictor-09 自己範疇說明中「重新驗證/擴充」的措辭），因為單純的 LOC/檔案數預估對這種節點形狀而言是個弱訊號——真正的成本驅動因素是對既有程式碼與已記錄假設進行推理的時間，而不是新增的表面積。
- 以實證方式確認（而不只是閱讀 `internal/storage/sqlite/db.go` 的註解）：這個套件的 SQLite WAL + `busy_timeout=5000` 設定，讓 `UPDATE ... WHERE consumed_at IS NULL` 的恰好一次保證,在相當大的範圍內與競爭程度無關：把並行重放測試的 goroutine 數量從 predictor-09 原本的 8 個提高到 64 個，不需要任何程式碼變動,測試的不穩定度也沒有增加。建議把「在既有的並行測試上把 goroutine 數量提高 8 倍、不改任何程式碼、確認結果依然成立」當作一個便宜、標準的強化階段技巧，用在未來任何恰好一次/冪等性稽核節點上，而不要假設一個已經通過的低並行測試,在更高的競爭程度下也足以構成充分證據。
- 一個共用的 `requireDomainError(t, err, wantCode) *domain.Error` 測試輔助函式（把斷言過的錯誤回傳給少數需要進一步檢查它的呼叫點，例如檢查 `Retryable`）取代了在重寫過的測試檔案中重複出現的 `errors.As`/`domain.Error` 樣板程式碼——但這在沒有把回傳值當作單獨陳述句使用的呼叫點上，引入了一個真實的 `golangci-lint` `errcheck` 發現。建議任何「只有部分呼叫端需要用到回傳值」的共用測試輔助函式，從一開始撰寫時就把這一點納入考量（例如記載清楚，作為單獨陳述句使用的呼叫端必須寫成 `_ = helper(...)`），而不是等到 golangci-lint 跑完才發現這個 lint 缺口——這裡因此多花了一次編輯。

## predictor-11 驗證做法：證明稽核什麼都沒找到，而不只是宣稱如此

不同於 predictor-10（Wave 8）找到並修正了一個真實的跨呼叫 bug，這個節點的 fail-open/fail-closed 對抗性測試（`TestFullPipeline_UpstreamErrorsFailClosed_NeverSilentAllow`、`TestFullPipeline_StageErrorsFailClosed`、`TestFullPipeline_DegradedRunwayNeverSilentlyAllowsWhenEmergency`、`TestFullPipeline_NeverProducesPolicyRunFromMissingCriticalSignals`）發現既有的鏈路在每一個被注入失敗的地方都本來就是正確的。依照 Constitution §6/§7 的紀律（「完成意味著有證據」，而不是「宣稱」），一個否定性的稽核結果,只有在它是可被否證的情況下才有意義——這套測試套件就是這樣：每一個 DataSource 方法,以及全部四個 ADR-041 pipeline-stage 介面,各自都有專屬的 error-injection 路徑（九個 `DataSource` 方法、四個 stage wrapper），每一個都透過一個真正的 `evaluation.Service` 接線（不是對 service 本身做 mock），每一個都斷言具體的安全結果（`EvaluateTurn` 回傳一個錯誤,「而且」在 `predictions` 表中該 TurnID 留下零筆資料——這是直接用 SQL 檢查的，不只是檢查「err != nil」）。如果這套測試套件只檢查「err != nil」而不同時確認零部分持久化，那會是嚴格意義上更弱的版本，因為 CONTRACT_FREEZE.md 的 transaction-boundary 章節,把部分持久化（而不只是錯誤的回傳值）指名為一個 `WithTx` 包裝過的操作必須防止的具體失敗模式。

## predictor-11 跨節點觀察：由寬表驅動的全鏈路模糊測試沒有抓到任何新東西，而這件事本身就有意義

`TestFullPipeline_WideTableFuzz`（11 個手選的 fixture，涵蓋冷啟動/完整填入/病態數值/極端量級/runway 緊急/runway debounce 等形狀，再加上 200 組以程式方式隨機產生的輸入組合，橫跨 quota/context 的 UsedPercent 範圍——包括超出 [0,100] 的數值——以及 0 到 40 筆樣本的 token 歷史）在整條 Scope->Token->Quota->Risk->Policy->Decide 鏈路上,找到零個 panic、零個 NaN/Inf 逸出、零個無效的 Confidence 值。這印證了（而不只是重複了）每一個個別 stage 自己先前的 property 測試（predictor-04/-06/-07/-08 各自的「no NaN/Inf」掃描）：在每個 stage 已經各自獨立證明過這個性質的情況下，在鏈路層級沒有發現「新的」失敗，這證明了各個 stage 的 clamp/降級紀律（`clamp01`/`clamp01Risk`、「未知不等於零」的指標語意、冷啟動 `Calibrated=false` 的閘門）在交接之間也能正確地組合起來，而不只是在單一 stage 自己的邊界之內成立。這是套件層級的測試套件無論再怎麼周全,都無法單靠自己提供的那種特定信心，而且很可能對未來任何多階段的 Auspex pipeline（不只是這一個）該如何收尾它的最終整合節點,提供有價值的通用指引：即使每一個個別 stage 都早已有自己的 property 測試，一個廣泛、對抗性、鏈路層級的 property fuzz 測試仍然值得跑，正因為交接處的 bug 就設計本質而言,對逐 stage 的測試是不可見的。

## predictor-11 效能基準測試結果 vs. ADD §29.11

`BenchmarkEvaluateTurn_FullPipeline`（一次完整的 `EvaluateTurn` 呼叫：Scope -> Token -> Quota -> Risk -> Policy，加上 `feature_vectors`/`predictions`/`policy_decisions` 的 transaction 持久化，對著一個真正跑過 migration 的 SQLite DB，`benchtime=1000x`，不含 `-race`）：**每次操作約 98 微秒、約 6.8 KB、136 次 alloc**。`BenchmarkEvaluateTurnThenDecide_FullPipeline`（相同的測試，再加上緊接其後的 `Decide` 讀回動作——依照 `internal/orchestrator/evaluate.go` 既有的接線方式，一個真實呼叫端一律會執行這個動作）：**每次操作約 189 微秒、約 12 KB、310 次 alloc**。在 `-race` 之下（這個節點自己要求的驗證指令），兩組數字都大約放大了一個數量級（分別約為 1.3ms 和 3.4ms/op），純粹來自 race detector 的插樁開銷，而不是來自正式程式碼路徑本身。跟 `Auspex_ADD.md` §29.11 所訂的目標——**warm evaluate P50 < 25 ms，P95 < 100 ms**——相比，即使是放大過的 `-race` 數字，對 P50 目標也還有大約 20-70 倍的餘裕，對 P95 目標也還有大約 30-75 倍的餘裕；非 `-race` 的數字則有大約 130-250 倍的餘裕。`predictor-08` 自己的 `BenchmarkDecide`（單獨的 Policy，依機器負載而定約 53-128ns/op）仍然遠低於獨立的 `policy < 1ms` 子預算，這個 phase 並沒有改變這一點。建議未來的角色在對一個由 SQLite transaction 支撐的熱路徑做效能基準測試時，一律把 `-race` 和非 `-race` 的數字並列，對照 ADD 所訂的預算一起回報，因為這兩者可能相差一個數量級，如果審查者只拿 `-race` 的數字去對照一個很緊的外部預算，可能會誤以為存在一次真實的效能回歸。

## predictor-11 角色回顧：橫跨 Wave 1-9 的完整歷程

這是 predictor 角色被指派的最後一個 DAG 節點——從 `predictor-01` 到 `predictor-11` 的每一個節點現在都已經是 `completed`，這個角色的完整範疇（`internal/features/**`、`internal/predictor/**`、`internal/policy/**`、`internal/evaluation/**`）就此收尾。回顧全部九個 phase：

- **這條 pipeline 是靠合約修訂長出來的，不是靠悄悄的範疇蔓延。** ADR-041（在任何 Wave-2 程式碼寫出來之前就被接受）把原始 Bootstrap DAG 混為一談的東西（`predictor-07` 依賴 `predictor-06`）拆解成正確的五階段鏈路（Scope -> Token -> Quota -> Risk -> Policy，Runway 獨立），再加上兩個新節點（`predictor-05b`/`predictor-05c`）。之後每一個 phase 都是直接依照這個修正過的合約建構的；沒有任何後續的 phase 需要回頭修補一個結構性的錯誤，因為這個錯誤在實作開始「之前」，就已經在合約層被抓到並修正了——正是 Constitution §6/§7「在程式碼寫出來之前發現的真實缺口，要在合約層修正」這條原則所預期的樣子。
- **在整段歷程中，總共發現了兩個各自獨立的真實 bug，而且都是靠專門為高風險節點所寫的對抗性測試找到的，從來不是靠偶然發現**：predictor-08 未經 clamp 的 NaN/Inf RiskScore 洩漏（Wave 6，被那個 phase 自己要求的 fail-open/fail-closed 測試抓到），以及 predictor-10 的 prompt-hash 綁定繞過（Wave 8，被一次專門的對抗性稽核找到）。predictor-11 自己的對抗性測試（Wave 9）沒有找到第三個 bug——這是一個正當、形態不同但同樣嚴謹的結果，而不是一個較遜色的結果，前提是（如上所述）這個否定性的結果本身就是可被否證的證據，而不是「沒去找」。
- **每一個需要自行發明 bootstrap 常數的 stage，都確切記錄了為什麼、以及怎麼發明的**（predictor-05 的冷啟動 fallback 表、predictor-05b 的 token 乘數上限與 P80 內插、predictor-05c 的預設 quota/context delta、predictor-06 的離群值門檻）——這些沒有一個被悄悄宣稱成實測數值；每一個都可以追溯到「ADD 指名了機制，但沒有指名這個具體常數」，並且標記為一旦後續 phase 有真正的歷史遙測資料落地,就要被取代。這項紀律,連續九個 phase 一致地被套用，正是這樣才讓 predictor-11 的跨 stage 稽核可以完全專注在交接的正確性上，而不必重新爭論任何個別 stage 的數字是否站得住腳——那個問題早已在每個 stage 被建構出來的當下,逐 phase 得到了解答。
- **冷啟動安全、未校準不等於機率這個不變量（Constitution §6/§7，這個角色最具承載力的一條規則），在每一個被測試到的地方、在每一層都成立**：`domain.RiskComponent`/`TokenForecast`/`QuotaForecast` 自己的 `Calibrated` 欄位、`policy.Decision.Probability` 那個只有兩處呼叫點的結構性防護，以及現在這個 phase 的全鏈路 fuzz 測試和 fail-open/fail-closed 測試套件，都確認了同一個性質透過真實持久化的 `Evaluation` 端到端成立。沒有任何一個 phase 曾經發現過對這個特定不變量的違反——最接近的一次是 predictor-08 的 NaN/Inf 洩漏，那是一個相關但不同的 bug（一個分數量級的 bug，不是一個校準宣稱的 bug）。
- **每一個 phase 都先合併 `origin/main`，並在寫任何新程式碼之前確認整個 repo 的 build/test 是乾淨的**，及早抓住整合上的漂移，而不是等到後續合併時才發現。這個 phase 的合併（`379b7cf` -> `36e7ffb`，Wave 8 的整合狀態）是乾淨的，在 predictor-11 自己的工作開始之前不需要任何調整，與這個分支上之前每一個 phase 的經驗一致。
- **給未來任何要建構類似多階段 predictor/policy pipeline 的專案的建議**：保留一個明確、獨立於逐 stage 建構節點之外的最終「證明整條鏈路」節點（就像這個專案的 `predictor-11`），即使每個 stage 都早已有周全的套件層級測試——它的價值不在於重複的覆蓋率，而在於交接本身的覆蓋率，而交接就設計本質而言,對任何單一 stage 自己的測試套件都是不可見的，而且這個 phase「零個新 bug」的結果之所以有意義，正是因為這個獨立的節點存在，並且被要求遵守跟「找到 bug 的 phase」相同的證據標準。

### 「是真實查詢還是冷啟動」的判斷才是真正的工作，不是 SQL 本身

不同於典型的 predictor-0N 節點（依照一個已凍結的介面實作一個公式），這次修正的九個方法各自都需要一次獨立的判斷——閱讀介面自己的文件註解和實際的 schema，然後誠實地判斷究竟有沒有一個真實的查詢存在，還是誠實的答案就是 `ok=false`——沒有單一先例可以機械性地套用在全部九個方法上。其中兩個判斷,結果證明在性質上跟其他七個真的不一樣：

- `RecentSimilarTurnTokens` 和 `RunwayForecast` 都是「真實的查詢,但實務上今天誠實地是空的/不存在的」——這個查詢本身是正確的，一旦上游資料存在（分別是未來的 claude-provider payload 欄位、未來的 `internal/pause` 持久化 phase），就會自動啟用，這正好呼應了 predictor-05b 自己「>=8 樣本的分支是真實的，但這個 phase 還無法觸及」的先例。
- `Repository` 和 `Session` 則是「再怎麼查也沒有用」——這兩個 DTO 所承諾的每一個欄位，描述的都是一種訊號類別（repository 檔案/語言普查、經驗性的 session-turn quantile），而在這次修正所限定的範疇內的 schema 中，完全沒有任何持久化的表示方式，而如果硬要合成一個出來（例如從 `turn.failed`/`turn.started` 的計數猜測一個「重試」的定義），那等於是打著「實作一個介面」的名義，發明一個新的、未經記錄的建模決策——建議未來遇到類似「介面承諾了 N 個欄位，但只有一部分有真實資料支撐」情況的角色，在報告中明確區分這兩種情況（就像這次修正所做的那樣），因為把兩者混為一談,要嘛會誇大涵蓋範圍，要嘛會低估所花的心力。

### 再一次，把其他角色 migration 檔案的註解當作有承載力的文件來讀

與 predictor-09 自己在 Wave 7 的心得（「migration 檔案……兼具了有承載力的設計文件功能」）一致，這次修正中 `RunwayForecast`/`PriorRunwayHitConfirmed` 的實作，對 `0042_runway_forecasts.sql` 和 `0043_policy_decisions.sql` 自己的標頭註解（記載了 turn_id 而非 session_id 的範疇界定，並交叉標註了哪個下游角色/方法會消費每一張表）的依賴程度，不亞於欄位清單本身——建議整個專案持續採用這種方式撰寫 migration 標頭，因為這已經是第二個獨立的案例，證明一份 migration 自己的文字敘述（而不只是它的 DDL），才是下游實作者做設計決策時正確的主要來源。
