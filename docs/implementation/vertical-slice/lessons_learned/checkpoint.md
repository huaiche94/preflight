## Overall retrospective: the checkpoint role's full arc (Wave 1 - Wave 9, checkpoint-a01 through checkpoint-b09)

This is the role's last assigned DAG node (`checkpoint-b09`); no further
`checkpoint-*` nodes remain in `EXECUTION_DAG.md`. Nineteen nodes shipped
across nine waves (a01-a09, b01-b09), plus one corrective fix
(`corrective-qa05-p1`). A few patterns held consistently enough across the
whole arc to be worth recording once, at this altitude, rather than only
per-node:

1. **Two independently-testable sub-components stayed independently
   testable the whole way through.** Part A (`internal/progress`,
   `internal/statecheckpoint`, `internal/artifacts`) and Part B
   (`internal/gitx`, `internal/repocheckpoint`, `internal/redact`) never
   needed to import each other, exactly as `agents/checkpoint.md`'s
   "Cross-part boundary" section mandated from day one. This was not free —
   it required deliberate injected-interface seams (`TreeReader` in
   `statecheckpoint`, the `EventPublisher` seam in `progress`, the
   `resolveWorktree`/`WorktreeLocation` callback in `repocheckpoint`) rather
   than the more convenient direct import — but it paid for itself directly
   in `checkpoint-a09`, where proving the two Reconcilers agree about the
   same crash window depended on them being two genuinely separate,
   independently-built pieces of logic rather than one shared abstraction
   that could have silently agreed with itself.

2. **A prior node's own scope boundary reliably predicted the next node's
   real, narrow work — five separate times** (`b04`->`b05`, `b06`->`b07`,
   `a04`->`a05`, `a05`->`a08`, and `a04`/`a06`->`a09`'s own "prove the
   pieces compose" framing). Every one of those "verify what remains"
   nodes came in at or under its nominal size once the real scope was
   correctly identified from the earlier node's own notes, rather than
   independently re-derived from a blank slate. The lesson generalized: read
   the immediately-preceding node's own lessons-learned/assumptions section
   as a literal scope prediction before assuming a fresh design phase is
   needed.

3. **The same fixture-quality failure mode recurred three times** across
   the whole arc: `checkpoint-a04`'s test-double `seqIDGenerator` race
   (Wave 6), `checkpoint-b06`'s too-short synthetic Azure-key fixture (Wave
   6), and `checkpoint-a09`'s identical-content-across-nodes bug (Wave 9,
   this file's own row above). All three are the same underlying mistake —
   a fixture that is too uniform, too easy, or not concurrency-safe silently
   under-tests the exact invariant the test claims to prove, and the test
   still runs green (or, in a09's case, fails for the WRONG reason, catching
   itself). Recommend this be promoted from "a lesson learned three times"
   to a named, explicit review question for any future role's fixture
   design in this codebase: "does every entity in a multi-entity test
   genuinely differ from its siblings in the dimension being tested, and is
   every test-double genuinely safe under the concurrency mode the real
   production code supports?"

4. **Empirical verification before trusting a design, not just code
   review, found every one of this role's non-obvious bugs** — the
   `seqIDGenerator` race (found by `-race`, not by reading), the `git apply
   --check` base-state semantics (found by a scratch-repo reproduction, not
   by reading the ADD), the Azure-key fixture (found by a disposable sanity
   script), and, most consequentially, `checkpoint-b09`'s manifest
   path-traversal vulnerability in `Verify` (found by writing and running a
   standalone proof-of-concept that actually read a file outside
   `ArtifactRoot`, not by reasoning that the code "looked risky"). This
   role's single strongest recurring practice, worth stating plainly for
   whichever role or wave comes next: when a design or a security property
   can be verified by actually running something, run it — before
   committing to either "this is fine" or "this is broken."

5. **The Part A/Part B split's two "final gate" nodes turned out to have
   genuinely different centers of gravity**, matching their own DAG risk
   annotations exactly: `checkpoint-a09` (integration risk) found zero
   production bugs and produced pure proof of already-correct composition;
   `checkpoint-b09` (security risk) found one real, previously-unknown,
   exploitable vulnerability and required an actual production fix before
   the proof could be honest. Both outcomes were legitimate for their
   respective risk framing — an integration gate's job is proving
   composition holds, and it can honestly conclude "everything was already
   correct"; a security gate's job is finding what's wrong, and a
   security-gate node that finds nothing should treat that as a reason to
   audit harder, not as confirmation of safety. `checkpoint-b09` did not
   stop at the DAG's own named required-test categories once it started
   finding real gaps — it kept auditing every path-join in its owned
   packages until it ran out of plausible untrusted-input surfaces, which is
   the posture a security gate this role would recommend for any future
   node with an equivalent risk annotation.

No further recommendations beyond what is already recorded per-node above;
this section exists only to name the patterns that recurred across the
whole arc rather than within any single node.

6. **"Every individual piece is tested" is not the same claim as "the
   frozen interface has a real implementation," and nothing in this role's
   own a01-a09 checklist would have caught the difference.** The Final
   integration gate found that `app.ProgressTreeService` — the actual
   contract `cmd/auspex/main.go` needs to compose the app's root — had
   no concrete type behind it anywhere in the repository, despite every
   underlying piece (`NodeStore`, `EdgeStore`, `ArtifactStore`, the state
   machine, `CompleteNode`, `Reconciler`) being exhaustively tested across
   nine waves, including `checkpoint-a09`'s own "full stack holds together
   end-to-end" integration gate. `a09` proved the pieces compose against
   each other; it never asked "does anything satisfy the actual frozen Go
   interface a caller outside this package would use." The gap survived an
   integration gate explicitly designed to catch composition problems
   because the gate's own test harness (`newFullStackHarness`,
   `complete_node_integration_test.go`) wires the pieces together by hand
   inside the test, which proves the pieces CAN compose but is not the same
   artifact as a production type that DOES compose them behind the frozen
   port. Recommendation for any future multi-piece role with a frozen
   interface at the end of its DAG: add an explicit, named checklist item —
   separate from "do the pieces work together" — for "does a
   `var _ app.X = (*ConcreteType)(nil)` assertion exist for a real,
   production type, not only a test double." A role can pass every one of
   its own DAG nodes and still leave this gap, because no single node's
   brief happened to ask the question directly.
7. **The fix itself confirms the DAG's own repeated framing was right: this
   was composition, not new logic.** `internal/progress.Service` (the
   corrective addition) is ~250 lines, none of which reimplement a state
   transition, an atomicity guarantee, or an idempotency check — every
   method is a DTO translation plus a direct call into an already-tested
   piece. The one substantive judgment call was `FailNode`'s
   `FailureClass`, which the frozen `FailNodeRequest` DTO accepts but which
   has no persisted column in this role's owned `0020-0029` migration
   range; rather than silently dropping it or inventing a column outside
   the corrective addition's stated scope, it is validated as non-empty and
   documented as an explicit, narrow gap for a future node — the same
   "surface the gap explicitly rather than paper over it" discipline this
   role's Constitution obligations (§7.3) already required for provider
   capability gaps, applied here to an interface/schema gap instead.
