## 0. 方法與誠實性約束

以下每個數值都標註為下列其中之一：**觀測（Observed）**（由主導者（lead）透過 `git diff --stat`、`git show` 或 harness 回報的用量區塊直接量測）、**估計（Estimated）**（由執行代理自行回報、未經獨立量測——例如來自一個無法存取時鐘的模型所稱的「約 20 分鐘牆鐘時間」）、**推導（Derived）**（由其他數值計算得出），或**未知（Unknown）**（不存在任何資料；並非捏造）。

有兩項結構性事實約束著整份報告，並在每一列中反覆出現：

1. **凍結的 DAG（`docs/implementation/vertical-slice/EXECUTION_DAG.md`）在任何地方、對任何節點都沒有工期欄位，也沒有 token 欄位。** 因此下方每一個 `estimated_duration` 與 `estimated_token_usage` 儲存格都是 `Unknown`——不是一個很小的數字，也不是零。這是本報告最重大的單一發現，並在 Phase 3.6 中詳細討論。
2. **沒有任何隊友為任何節點擷取「讀取檔案數」指標。** 只有「變更檔案數」（寫入／修改的檔案）有被追蹤。工具呼叫次數（harness 用量區塊中的 `tool_uses`）確實存在，但混雜了讀取、寫入、編輯與 shell 指令——若將其當作「讀取檔案數」的替代指標來回報，等同於換個標籤的捏造。每個節點的 `actual_files_read` 皆為 `Unknown`。

實際 token 用量僅在**每隊友每波次的顆粒度**上為 `Observed`（每次背景代理呼叫一個數字，涵蓋該次呼叫所完成的所有節點），從未達到每節點的顆粒度，因為子代理並未自行回報每節點的 token 明細，harness 也不提供。凡是某節點與同批節點共用一個隊友波次 token 總量之處，皆明確述明，而非平均攤分後當成每節點資料呈現。

## 3. 本報告「不」主張的事項

- 本報告不計算 token 用量誤差，因為從未存在可供比較的 token 估計值（§0）。此處出現的任何數字都會是捏造。
- 基於同樣的理由，本報告也不計算耗時誤差。
- DAG 中的 LOC「估計值」是一個沒有文件化定義的單一數字（毛插入行數？淨行數？僅實作，或實作 + 測試？）。本報告將其視為可與 git 的「insertions」計數相比較——這是最接近的可用類比——但這是一項詮釋性選擇，而非既定的等價關係——在此明確標示，而非默默假定其完全相等。
- 複雜度「誤差」不以數值量化（XS/S/M/L/XL 是次序性標籤，不是基數尺度）——§1 回報標籤是否維持不變，而 Calibration_Report.md（Phase 3.2）則討論其方向。
