本報告不會重複 Feature Registry 的身分／來源（provenance）欄位（比照 Constitution 的規則：不重複規範性的資料字典）。本報告新增的，正是登記表（registry）中沒有記載的內容：每個缺口為何存在、其影響、建議的補上方式、複雜度估計，以及預期的改善程度——並依優先順序排列，供未來的 wave 排定優先次序。

## 以下使用的複雜度量表

為求一致性，沿用 DAG 自身的 XS/S/M/L/XL 慣例（`EXECUTION_DAG.md` §1），此處套用在*補上此缺口*這個工作單位上，而不是 DAG 節點估計值——這些並非新的 DAG 節點，只是供 Wave 3 以後規劃參考的複雜度訊號。

## 1. 重大重要性缺口

### 1.1 儲存庫特徵連接（Registry §2）

- **缺失原因：** 不是資料缺口——是連接缺口。`internal/gitx` 已經計算出 dirty 檔案／行數計數與 worktree 結構（真實且已測試）。`predictor-05` 的 `FeatureSource.Repository()` 方法沒有具體實作，只有一個測試用的假物件（fake）。
- **影響：** `ScopeEstimate.FilesChangedP*`／`LinesChangedP*`——Rule Predictor 最重要的兩項輸出——目前從未看到真實的儲存庫狀態，只看到冷啟動預設值與工作階段歷史的混合值。
- **建議實作方式：** 在未來的 predictor wave 中，實作一個具體的 `FeatureSource`，呼叫 `internal/gitx` 已建置好的基礎功能（primitives）。不需要新的 gitx 功能——這只是膠合程式碼（glue code）。
- **複雜度：** S——困難的部分（Git 內省）已經完成；這只是轉接器／連接工作。
- **預期的預測改善程度：** 高。這是整個 registry 中成本最低、槓桿效益最高的單一缺口：真實資料只差一次函式呼叫就能取得。

### 1.2 實際 token 使用量／turn（Registry §4、§5 的 `TokenForecast`）

- **缺失原因：** 從未有任何真實的編碼代理工作階段針對 `auspex` 執行過（目前還沒有可以觀察工作階段的 CLI）。
- **影響：** 在每一個層級（Rule／Statistical／ML）都永久阻擋 Token Forecaster 的校準——無論演算法多精密，都沒有可供校準的真實基準（ground truth）。
- **建議實作方式：** 需要 `runtime-b01` 以後（CLI／hook 整合）以及 `foundation-06`＋`claude-provider-05`（持久化）——這不是單一缺口，而是一條相依鏈。
- **複雜度：** L——橫跨三個角色份量、尚未建置的基礎設施，不是單一節點。
- **預期的預測改善程度：** 重大，但無法漸進式實現——這是一個全有或全無的解鎖點（依 ADD §15.6 的 ≥20 樣本門檻，一個只有部分遙測資料的 Token Forecaster 仍然是未校準的）。

### 1.3 State Checkpoint／產物（Artifact）產生者（Registry §6）

- **缺失原因：** Progress Tree 與 State Checkpointing（`checkpoint-a01` 至 `-a09`）完全尚未建置——卡在 `foundation-06` 上。
- **影響：** 並非直接的預測品質缺口（依 Registry 自身的註記，Checkpoint 特徵並不會餵入 predictor），但會使 Constitution 自身「完成即代表有證據」的不變性原則，完全沒有任何持久性的落實機制。
- **建議實作方式：** 已完整規格化為 DAG 節點 `checkpoint-a01`-`a09`；不需要新的規格工作，只需要在 `foundation-06` 解鎖後執行即可。
- **複雜度：** XL——`checkpoint-a04`（`CompleteNode` 原子協定）在本專案自己的其他文件中，已被獨立標記為整個 DAG 中影響最深遠的單一任務。
- **預期的預測改善程度：** 直接來看沒有（見上方「影響」）——之所以列為重大重要性，是基於另一個原因（產品完整性，而非預測準確度），會列在此處是為了 registry 的完整性，而不是因為補上此缺口能改善校準。

## 2. 高重要性缺口

### 2.1 Provider 特徵從僅限 fixture 範圍轉為真實資料（Registry §4）

- **缺失原因：** `claude-provider` 的解析器／正規化器（normalizer）是真實存在且測試正確的，但至今只解析過手動建構的 JSON，從未解析過實際 Claude Code 行程所發出的任何位元組。
- **影響：** 每一個 `UsageObservation`／`QuotaObservation`／`ContextObservation` 的下游使用端，都繼承了相同的「在 fixture 上證明正確，但在真實情況下未經驗證」狀態。這是目前程式碼庫中「看起來完成、但未經真實世界驗證」風險最集中的地方。
- **建議實作方式：** 一次真實環境的整合冒煙測試（smoke test）——執行一次真正的 `claude` CLI（或其受控代理），擷取真實的狀態列／hook 輸出，並確認現有的解析器不需修改即可處理。這不需要先建好完整的 CLI；可以是一次範圍窄、一次性的驗證作業。
- **複雜度：** M——主要在於如何安全地取得真實樣本，而不是撰寫新程式碼（解析器已經存在）。
- **預期的預測改善程度：** 本身不會改善任何預測數字，但能在更多基礎設施建立於其上之前，把一個未經驗證的假設轉為已驗證的假設——這是一個降低風險的缺口，而非準確度缺口。

### 2.2 工作階段特徵（Registry §3）

- **缺失原因：** 完全依賴 A5（`Missing_Telemetry_Report.md`）——尚未存在的歷史結果／使用量資料。
- **影響：** 阻擋整個 Version 2「Statistical Predictor」層級（Design Supplement 自身路線圖中的項目）——這個特徵群組的既定用途，正是統計型 predictor 所需要、而規則型 predictor 並非嚴格必要的東西。
- **建議實作方式：** 沒有捷徑——必須先累積真實的 turn 資料。
- **複雜度：** XL（與 1.2 相同的相依鏈，並非獨立來看會比較小）。
- **預期的預測改善程度：** 重大，但只有在 1.2 補上之後才能實現——這兩個缺口共用同一個根本的阻塞點。

## 3. 中重要性缺口

### 3.1 `ProviderCapabilities` 的真實偵測（Registry §4）

- **缺失原因：** 沒有任何 `ProviderCapabilityReader` 的實作；該型別已在 Bootstrap 階段凍結，但能力偵測從未與任何 provider 串接。
- **影響：** ADD 中每一個以能力為門檻的行為（降級規則，ADD §8.8），目前都沒有任何執行期訊號可供把關。
- **建議實作方式：** 屬於 `claude-provider` 的交付項目——每個工作階段偵測一次版本／模式，並填入這個 19 個欄位的 struct。
- **複雜度：** S——範圍窄、規格清楚（ADD §8.6／§8.7 已明確列出要偵測的確切內容）。
- **預期的預測改善程度：** 對預測*準確度*的直接影響為低至中，但能有意義地降低因為能力是被假設、而非被確認，導致預測悄悄出錯的風險。

### 3.2 儲存庫相依關係圖擴散（fan-out）（Registry §2、ADD §14.4）

- **缺失原因：** 已在 ADD 中規定，但從未實作——此程式碼庫中目前沒有任何程式碼呼叫 `go list -deps -json` 或 `dotnet list ... reference`。
- **影響：** 一旦 `RiskCombiner` 建置完成，`blast_radius_risk` 中跨層／跨專案的項目，目前將沒有真實訊號可供參考。
- **建議實作方式：** ADD §14.4 已明確規定了具體指令與快取策略；這只是實作既有規格，不是新的設計工作。
- **複雜度：** M。
- **預期的預測改善程度：** 中——只縮小一個特定、已命名的風險項目，不是廣泛的準確度提升。

## 4. 低重要性缺口

### 4.1 已讀取檔案追蹤（Registry §1、§5）

- **缺失原因：** 在本專案目前的能力矩陣中，沒有任何 provider 事件會區分「讀取」與「變更」。
- **影響：** 若不針對每個 provider 進行 provider-adapter 層級的工作，`ScopeEstimate.FilesReadP*` 就會永遠停留在估計值，從未有真實基準可對照。
- **建議實作方式：** 依 provider 各自的工具呼叫形狀（tool-call-shape）啟發式方法（見 `Missing_Telemetry_Report.md` A3）——這是 provider-adapter 的工作，不是 predictor 的工作。
- **複雜度：** 每個 provider M（每新增一個 provider adapter 就要重複一次）。
- **預期的預測改善程度：** 低至中——`FilesReadP*` 是一個已命名的欄位，但目前本 wave 建立的任何下游公式都尚未使用它（會使用它的 Token Forecaster，本身也還沒建好）。

### 4.2 校準統計量（ECE／Brier）（Registry §7）

- **缺失原因：** 需要 ≥20 筆真實、已標記的樣本（ADD §15.6）——這是數量缺口，不是實作缺口。
- **影響：** 在整個專案範圍內，卡住從「風險分數」轉變為「已校準機率」的門檻。
- **建議實作方式：** 不存在、也不應該存在任何實作上的捷徑——ADR-026／033 明確禁止繞過此門檻。
- **複雜度：** N/A——這是資料量門檻，不是建置任務。
- **預期的預測改善程度：** 直接來看是 N/A；它是*用來衡量*每個已校準層級改善程度的指標，而不是一個本身會產生改善的功能。
