## 1. 為何本報告不包含 P50/P80/P90 準確度數字

第 3.5 階段要求回放「目前已收集的每一筆歷史遙測紀錄」，並將「目前的 Rule
Predictor」與「提議的 Rule Predictor」進行比較。此比較所需的兩項前提皆不存
在，若為了填滿要求的格式而回報捏造的數字，將直接違反本階段自身的治理規則
（「寧可未知，也不可捏造資料」）。精確陳述如下：

### 1a. 不存在任何歷史遙測紀錄

Auspex 自身架構中的「遙測紀錄」，是指由真實的程式碼代理（coding-agent）回合
所產生、並持久寫入 SQLite 的
`turns` / `turn_usage` / `quota_observations` / `context_observations` 資料
列（ADD §12.2）。目前尚不存在任何能夠產生或儲存此類紀錄的基礎設施：

- `foundation-06`（SQLite 遷移、核心 session/turn 資料表）— 尚未建置（已延
  後至 Wave 2 之後／被阻擋；`foundation-05` 刻意只建置了引擎本身，沒有任何
  遷移檔）。
- `claude-provider-05`（將正規化後的遙測事件冪等地持久化至 SQLite）— 尚未
  建置（被 `foundation-06` 阻擋）。
- `predictor-09`（評估結果持久化）— 尚未建置。
- 從未有任何 `auspex` 執行檔觀察過真實的 Claude Code 回合。Wave 1／Wave 2
  期間*建置 Auspex 本身*所執行的工作，並不屬於此處所指的遙測——那是建置預
  測器的工作，而非預測器所產生的輸出。

沒有任何紀錄可供回放。這並非一個很小的數字，而是「不存在」，因此在本文件中
一律回報為 `Unknown`／`N/A`，而非 `0`（若標示為 `0`，會誤導讀者以為存在一個
可運作的管線、只是其結果集恰好為空，但事實上目前根本不存在任何能產生此類紀
錄的管線）。

### 1b. 不存在可與「目前版本」比較的「提議版 Rule Predictor」

第 3 階段明確禁止在本分析階段修改預測器的係數或實作（3.2 節與 3.4 節皆有此
聲明）。在 Wave 1、Wave 2 或本分析階段中，均未建置任何 Rule Predictor 的替
代／提議版本。目前僅存在唯一一種實作——`RuleScopeEstimator`（`predictor-05`）
與 runway 用的 `Scorer`（`predictor-06`）——沒有任何對照組可供比對。「目前
版本 vs. 提議版本」的比較需要有兩個對象可供比較；但目前只存在一個。

## 3. 確實存在的：內部自我一致性證據（並非準確度）

為避免本報告全然是負面內容，以下是本波（phase）中真實、已觀測到的證據——但
這明確是一個與「預測準確度」不同、且較弱的主張，讀者不應將兩者混為一談：

- `predictor-05` 的測試套件（`internal/predictor/scope/estimator_test.go`，
  已由主導者獨立重新執行並驗證）確認在涵蓋冷啟動（cold-start）、混合式
  session 歷史紀錄、以及明確檔案路徑等情境的 8 個具名情境中，
  `P50 <= P80 <= P90` 皆成立。這是一項**結構性保證**，而非準確度量測——它
  證明輸出在結構上是井然有序的，但不代表這些有序的數值接近任何真實結果。
- `predictor-04` 的分位數（quantile）工具函式
  （`internal/predictor/quantile.go`，Wave 1）已透過 2000 次隨機試驗進行屬
  性測試（property-tested），從未違反單調性，也未產生 NaN/Inf。同樣的附帶
  說明：這證明*數學*本身是正確的，但不代表它已能預測任何真實事物，因為它
  從未餵入過真實的程式碼代理遙測資料。
- `predictor-06` 的 runway 評分器已針對約 300 種輸入組合進行掃描測試
  （`TestScoreNeverCalibratedNeverPanics`），並在缺乏持久化遙測資料的情況
  下，始終正確回報 `Calibrated: false` 與 `HitProbability: nil`——這是一種
  **拒絕誇大主張的正確行為**，正是《憲章》所要求的冷啟動安全行為，但這只保
  證了*預測器誠實地說出它不知道什麼*，並非對其正確之處的量測。

## 4. 未來哪些條件成立後，本報告才具備可產出性

此處僅列為前瞻性指引——並非建議現在就建置以下任何項目，也不是一份實作計畫：

1. `foundation-06` + `claude-provider-05`（持久化的遙測資料儲存）必須存在，
   且必須實際觀測到真實的 Claude Code 回合。
2. `predictor-09`（評估結果持久化）必須存在，讓預測結果本身能與其所預測的
   遙測資料一併被持久記錄下來。
3. 必須累積最低數量、結果已知且已完成的回合樣本（ADD §15.2 自身即以
   `count(similar) >= 8` 作為其內部退出冷啟動狀態的門檻——這對任何回放分析
   而言，都是一個合理的初始下限，不僅適用於即時預測）。
4. 唯有到那時，「目前版本 vs. 提議版本」的比較才具有意義；且僅在確實有意
   圖地撰寫出實際的第二個（提議版）規則變體、作為比較基準之後。

本主題已在 `Missing_Telemetry_Report.md`（第 3.6 階段）中以具名指標與建議實
作方式作更詳細的說明。
