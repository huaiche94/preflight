## A. 產品遙測（Auspex 預測器的輸入）

### A1. 每回合的實際 token 使用量

- **目前狀態：** 未知
- **資料來源：** 未知
- **無法取得的原因：** 從未有任何即時的 Claude Code（或 Codex）session 對
  `auspex` 執行檔運行過——CLI（`runtime-b01` 以後）尚不存在，因此沒有任何
  整合點能夠觀測到真實的回合。
- **供應商限制：** 依 ADD §8.7 自身的能力矩陣，即使完成整合，精確的每回合
  使用量對 Claude 原生 hooks/status 而言是「部分／由狀態推導」，對 Claude
  受管的 stream-json 而言則是「取決於串流」——並非在每種模式下都保證完整
  或精確。
- **可能的確定性替代方案：** ADD §14.7 的提示詞 token 近似公式
  （`ceil(ASCII/4.0) + ceil(CJK/1.5) +
  ceil(other-Unicode/2.5) + ceil(punctuation/3.0)`、`confidence=low`）——
  已有規格，但尚未接上任何即時輸入。
- **對預測品質的預期影響：** 關鍵。缺少此資料，Token 預測器
  （`predictor-05b`，尚未建置）將永遠沒有可供校準的真實資料（ground
  truth）——它做出的每一筆預測都會永久停留在 `Calibrated: false`。
- **建議的未來實作方式：** `claude-provider-05`（持久化遙測資料）加上即時
  的 status-line/hook 整合是直接前置條件；不存在任何捷徑。

### A2. 剩餘的五小時配額

- **目前狀態：** 估計值（僅在即時遙測資料存在後才成立；目前為未知——沒有
  任何即時 session 可供估計）
- **資料來源：** 未知（一旦滾動使用量帳本存在，將成為推導值，依 ADD §15.9
  的「Realistic Case」）
- **無法取得的原因：** 與 A1 的根本原因相同——沒有即時 session。
- **供應商限制：** ADD §8.7：配額的揭露方式，對 Codex 原生 hooks 而言是
  「sidecar App Server」，對 Claude 原生 hooks 而言是「僅限受支援的訂閱
  者」——即使在即時狀態下也非普遍可得。
- **可能的確定性替代方案：** ADD §15.9 的滾動使用量帳本
  （`Σ tokens within last five hours`），搭配由觀測到的限額事件估計、而非
  假定為固定值的 `effective_limit`——ADD 中已有規格，尚未實作（不存在可承
  載該帳本的持久化層）。
- **對預測品質的預期影響：** 對配額預測器（`predictor-05c`，尚未建置）與
  Runway（`predictor-06`，本波已建置，但目前僅以合成測試輸入演練過，從未
  接收過真實的配額觀測值）而言至為關鍵。
- **建議的未來實作方式：** `foundation-06`（`quota_observations` 的 SQLite
  遷移）是直接前置條件。

### A3. 真實回合中實際讀取的檔案（相對於變更的檔案）

- **目前狀態：** 未知
- **資料來源：** 未知
- **無法取得的原因：** 本專案能力矩陣（ADD §8.6-8.7）中，目前沒有任何供應
  商遙測來源將「代理讀取的檔案」與「代理變更的檔案」區分為可計數的指標——
  僅有檔案變更事件（`FileChangeEvents` 能力）被具名列出。
- **供應商限制：** 供應商通常不會發出與工具呼叫日誌區分開來的結構化「檔案
  讀取」事件；此資料很可能需要從工具呼叫的酬載（payload）中推斷（例如一次
  `Read`/`cat`/`grep` 呼叫），而這取決於各供應商與各工具的結構描述。
- **可能的確定性替代方案：** 依各供應商解析工具呼叫事件，比對已知的讀取類
  工具名稱，作為啟發式的下限估計——尚未實作、尚未驗證。
- **對預測品質的預期影響：** 中高。`ScopeEstimate` 已具備
  `FilesReadP50/P80/P90` 欄位（已凍結，ADD §14.1），而 `predictor-05` 目前
  無法以真實資料填入這些欄位——它只能透過冷啟動預設值與 session 歷史混合
  來估計，從未依據真實資料，因為此欄位的真實資料從未存在過。
- **建議的未來實作方式：** 將供應商專屬的「工具呼叫對應到檔案讀取」的映射
  定義為各供應商轉接器（`claude-provider`、未來的 `codex` 轉接器）的一部
  分，而非放在預測器本身。

### A4. 真實程式碼代理回合的實際耗時

- **目前狀態：** 未知
- **資料來源：** 未知
- **無法取得的原因：** 從未端到端觀測過任何即時回合。
- **供應商限制：** 原則上預期沒有——回合起訖時間戳記是多數供應商都會提供
  的基本能力——這純粹是「整合尚未建置」的缺口，而非供應商的限制。
- **可能的確定性替代方案：** 一旦完成整合即無需替代方案；這是本報告中唯一
  一項不存在供應商端固有障礙的指標。
- **對預測品質的預期影響：** 高。`ScopeEstimate` 已具備
  `DurationP50`/`DurationP90` 欄位（ADD §14.1，已凍結），但在每一筆

- **建議的未來實作方式：** 一旦 `runtime-b01` 以後的節點存在、並觀測到真實
  回合的開始／完成事件，即可直接實作。

### A5. 歷史結果標籤（completed_normally / hit_usage_limit / required_compaction / user_interrupted / tool_failure / required_followup_turn）

- **目前狀態：** 未知
- **資料來源：** 未知
- **無法取得的原因：** 從未有任何回合在 Auspex 的觀測下完成（或失敗）；這
  些標籤（在 `Auspex_Predictor_Design_Supplement.md` 的「Better Statistical
  Models」一節中具名列出）需要一個已持久化、已終止的回合才能指派。
- **供應商限制：** 預期沒有——失敗／完成訊號在完成整合後通常都可觀測
  （`domain.FailureClass` 已凍結，且已具備所需的分類體系：
  `FailureProviderRateLimit`、`FailureTimeout`、`FailureUserInterrupt`
  等）。
- **可能的確定性替代方案：** 無需替代方案；一旦持久化層存在，這純粹是「尚
  未開始收集」的缺口。
- **對預測品質的預期影響：** 對任何 Version 2 以上（統計式／ML）的預測器層
  級而言至為關鍵——這些標籤正是《Predictor Design Supplement》自身路線圖
  中具名的訓練目標。目前的 Version 1 Rule Predictor 在運作上不需要它們
  （這是刻意設計，冷啟動安全），但缺少它們就無法演進至 Version 1 之後。
- **建議的未來實作方式：** `predictor-09`（評估結果持久化）加上
  `predictor-10`（授權機制，用於回合／決策的關聯）是直接前置條件。

## B. 流程遙測（實作 Auspex 本身）

### B1. 每節點的 token 使用量

- **目前狀態：** 僅在「每位隊員、每波」的粒度上有觀測值（19 個節點中有 10
  個與同批節點共用一次呼叫，無法拆分）
- **資料來源：** 觀測值（彙總）／未知（每節點）
- **無法取得的原因：** 執行框架（harness）對每次背景代理呼叫僅回報一個使用
  量總計，而非按工具呼叫或按節點細分；也未要求隊員自行回報 token 數（他們
  無法從對話內部觀測自己的 token 使用量）。
- **供應商限制：** N/A——這是執行框架／工具的能力缺口，不是產品意義上的供
  應商限制。
- **可能的確定性替代方案：** 未來各波中，讓每次背景代理呼叫恰好只分派一個
  節點，以平行化效率換取乾淨的每節點歸因——這是真實的取捨，不是免費的修
  正。
- **對預測品質的預期影響：** 直接阻擋 `Prediction_Error_Report.md` 對 19
  個節點中 17 個的每節點 token 誤差計算（該報告 §2）。
- **建議的未來實作方式：** 或者由執行框架提供隊員可自行回報的每回合 token
  數，或者 Wave 3 規劃時，在抽樣的一部分節點上刻意以部分平行度換取量測的
  乾淨度。

### B2. 精確（非自行回報）的節點耗時

- **目前狀態：** 19 個節點中有 10 個為未知；其餘 9 個為估計值（自行回報，
  未經時鐘量測）
- **資料來源：** 充其量為估計值
- **無法取得的原因：** 沒有提供任何隊員牆鐘時間戳記工具；「~20 min」這類數
  字是執行模型自身的近似，不是量測值。
- **供應商限制：** N/A
- **可能的確定性替代方案：** 由主導者記錄每次代理呼叫的生成時間戳記與完成
  通知時間戳記（主導者確實會透過任務通知收到真實時間戳記）——本波未系統性
  地執行此事，這是流程缺口，不是工具缺口。
- **對預測品質的預期影響：** 直接阻擋未來 DAG 中的任何耗時估計
  （`Calibration_Report.md` §8 將此列為最優先的改進項目之一）。
- **建議的未來實作方式：** Wave 3 中由主導者依其自身工具呼叫的時間戳記記錄
  呼叫起訖的牆鐘時間，與隊員的自行回報彼此獨立。

### B3. 每節點讀取的檔案數（流程範疇）

- **目前狀態：** 未知
- **資料來源：** 未知
- **無法取得的原因：** 與 A3 的產品範疇版本相同——沒有任何隊員或工具追蹤
  與「變更的檔案」區分開來的「讀取的檔案」計數。
- **供應商限制：** N/A
- **可能的確定性替代方案：** 執行框架的使用量區塊中存在 `tool_uses` 計數，
  但它混雜了 Read/Write/Edit/Bash——不是有效的替代品（見
  `Prediction_Error_Report.md` §0 明確拒絕將其重新標示為檔案讀取的代理指
  標）。
- **對預測品質的預期影響：** 對流程校準本身而言為低至中（檔案變更數才是與
  DAG 更相關的訊號）；對產品範疇的 A3 影響較大。
- **建議的未來實作方式：** 不列入優先——請見 A3 的產品範疇版本，其重要性
  更高。
