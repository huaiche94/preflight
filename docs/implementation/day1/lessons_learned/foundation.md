| foundation-02 | S | S (held) | 3 | 4 (paths.go, env.go, fake_env_test.go, paths_test.go) | 180 (DAG LOC estimate only, no duration unit) | ~20 min wall-clock | none — no new go.mod dependency needed, stdlib only (errors, path, runtime, strings) | fake_env_test.go as a separate file from paths_test.go (kept the test double out of the main test file for readability; DAG's "3 files" estimate landed close: it likely counted paths.go + env.go + one test file and didn't anticipate splitting the fake into its own file) | one real bug caught by the required cross-platform path-table tests: the first implementation used filepath.Join for the Windows branch, which uses the HOST OS's separator (produced forward slashes for Windows paths when run on this darwin dev machine) instead of the TARGET OS's separator being resolved. Fixed by writing a dedicated winJoin helper using literal backslashes and switching the POSIX branches to path.Join (always forward slash) instead of filepath.Join (host-dependent) — this is precisely the class of bug the DAG's own foundation-02 risk note ("Windows path behavior needs CI matrix") was pre-warning about; a fake-GOOS unit test caught it without needing an actual CI matrix | re-reading Preflight_ADD.md SS26.1/SS26.3 to confirm this package's scope boundary (global dirs only, not repo-local .preflight/**) took longer than writing the code itself — the ADD text names ".preflight/config.yaml" and "global user config" in the same precedence list without explicitly saying which package owns which, so scope had to be inferred from agents/foundation.md's wording ("OS-correct config/data/cache/runtime paths") plus Constitution SS7 rule 10 (don't build abstractions the current milestone doesn't need) rather than being stated directly | (1) filepath.Join-vs-host-GOOS is a subtle, easy-to-miss bug class specific to any package that resolves paths for a GOOS other than the one it's compiled/running on — worth calling out explicitly in agents/foundation.md or a shared Go style note for any future role doing similar cross-platform resolution, since `go vet`/`gofmt` do not catch it and it only surfaces via an OS-specific assertion in a test. (2) The DAG's foundation-02 risk column correctly predicted the failure mode in the abstract ("Windows path behavior needs CI matrix") but the actual fix needed no CI matrix at all, just a GOOS-parameterized Resolve() function and fake Env — future estimates could note that dependency-injecting the OS-selector itself (not just the environment) turns an "needs CI matrix" risk into a same-host unit test, which is strictly cheaper. |
